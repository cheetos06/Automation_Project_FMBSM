# FS Review Pipeline

This folder contains the active financial-statement review pipeline.

The intended flow is:

1. Render every N and N-1 PDF page as a gridded image. Page selection is fully
   automatic; no statement or annex page numbers are configured.
2. Send each physical page to Copilot exactly once. That canonical call returns
   the page scope, reviewable text blocks, Bilan/CPC lines and amounts, annex
   cells, and normalized visual bounds.
3. Distribute page calls dynamically across the configured Copilot accounts.
   Each account handles one request at a time, while the accounts run in
   parallel.
4. Refine Copilot's approximate bounds against visible pixels in the rendered
   page. This is local image geometry only: no OCR and no PDF text layer.
5. Derive `fs_extract_N.json`, `fs_extract_N_1.json`, `fs_tick_layout.json`, and
   the current/N-1 document indexes from the canonical records. This replaces
   the former overlapping extraction and position passes.
6. Align current and prior pages by document scope, keeping annual/company and
   consolidated statement sets independent.
7. Compare titles, paragraphs, annex tables, values, calculations, and evidence
   requirements using batched structured JSON calls. These calls do not resend
   page images.
8. Run the rule-based BG-to-FS and supported-annex matchers.
9. Write:
   - `BG_Mapped.xlsx`
   - `FS_Mapped.xlsx`
   - `mapper_audit.md`
   - `Document_Review.xlsx`
   - `financial_statements_N_ticked_by_pipeline.pdf`
   - `pipeline_report.json`

The FS extraction step must not use PDF text extraction or OCR libraries. The
Copilot runtime reused from `SOPs\OPTIMDA\Build 2` renders PDF pages to images
with PyMuPDF, uploads the rendered images, and asks Copilot to read the visible
content.

## Run With Copilot

From `STAT - BOUCLAGE`:

```powershell
python .\Pipeline\run_pipeline.py --ticket .\Examples\17779 --year 2025
```

This requires the Copilot session/configuration from:

```text
SOPs\OPTIMDA\Build 2
```

The worker accounts are selected by `copilot_account_names` in
`Pipeline\config\settings.json`. The default configuration uses `m365_work`
and `m365_isga`. Requests are independent and do not rely on conversation
memory, so distributing them does not change the extraction instructions or
review quality.

### Rate Limits And Cooldowns

The account pool applies the following rules to every canonical page and
structured-comparison request:

- A recognized Copilot throttle/HTTP 429 response immediately puts that
  account on a one-hour cooldown and requeues the same request for another
  available account. The transport does not retry a throttled request on the
  same account first.
- Each account may receive at most 150 dispatched turns before a one-hour
  cooldown starts. The 150th request is allowed to finish; subsequent work is
  routed to another account.
- If every configured account is cooling down, the pipeline waits until the
  first cooldown expires instead of dropping pending work.
- Turn counters and cooldown expiry timestamps are persisted in
  `config/copilot_pool_state.json`, so restarting the pipeline does not reset
  the limit. The counter resets when the cooldown expires.

The defaults are controlled by `copilot_turn_limit_per_account` (150),
`copilot_cooldown_seconds` (3600), and `copilot_pool_state_path` in
`config/settings.json`.

The normal command always scans all physical pages. Copilot identifies the
page type and scope from the image, and non-relevant pages return empty
financial/table collections while still contributing document wording where
appropriate. There are no ticket labels, account names, page numbers, or tick
coordinates hardcoded in the extraction or geometry logic.

The same command also performs the full wording and annex review. Use
`--no-document-review` only when intentionally running the Bilan/CPC mapper in
isolation.

## Measured 17779 Run

A clean run on the 2025 dossier `Examples\17779` processed 19 current pages and
15 N-1 pages:

- 34 Copilot vision calls: one for each of the 34 physical pages
- 5 Copilot structured comparison calls: 15 aligned page groups in batches of 3
- 39 successful Copilot calls in total, versus 73 in the previous architecture
- 10 minutes 47 seconds end to end on the two configured accounts
- 85/85 nonzero BG rows mapped
- 17/20 nonzero FS leaf lines justified, identical to the prior manual-quality
  mapper run
- 165 tickmarks written to the reviewed PDF

The call count fell by 34, approximately 47%. Actual time can vary with
Copilot response time and authentication retries.

## Annual And Consolidated Accounts

If one PDF contains both annual/company and consolidated accounts, provide one
BG per scope:

```text
Input/
  bg_standardized_annual.xlsx
  bg_standardized_consolidated.xlsx
```

The filenames can instead be set in `config/settings.json` under
`scope_bg_files`. The pipeline refuses to guess that one generic BG belongs to
both scopes. Scope outputs are written independently and their tickmarks are
combined only on the final PDF.

See `DOCUMENT_REVIEW_RULES.md` for wording, evidence, year-specific, and
missing-document rules.

## Isolated Steps

The canonical visual pass is intentionally orchestrated by `run_pipeline.py`
so page extraction is shared by mapping, document review, and tick placement.
The commands below remain useful for diagnosing individual legacy contracts or
rerunning local mapping/output generation; they are not the normal full-pipeline
path.

Use these when debugging one piece at a time:

```powershell
python .\Pipeline\extract_fs_only.py --pdf .\Examples\17779\Input\financial_statements_N.pdf --output .\Examples\17779\Output\extraction\fs_extract_N.json --prompt-key fs_extract_prompt

python .\Pipeline\extract_fs_only.py --pdf .\Examples\17779\Input\financial_statements_N_1.pdf --output .\Examples\17779\Output\extraction\fs_extract_N_1.json --prompt-key prior_extract_prompt

python .\Pipeline\align_prior_to_fs.py --prior-json .\Examples\17779\Output\extraction\fs_extract_N_1.json --current-fs-json .\Examples\17779\Output\extraction\fs_extract_N.json

python .\Pipeline\extract_layout_only.py --pdf .\Examples\17779\Input\financial_statements_N.pdf --output .\Examples\17779\Output\extraction\fs_tick_layout.json

python .\Pipeline\align_layout_to_fs.py --layout .\Examples\17779\Output\extraction\fs_tick_layout.json --fs-json .\Examples\17779\Output\extraction\fs_extract_N.json

python .\Pipeline\map_bg_to_fs.py --ticket .\Examples\17779\Input --fs-json .\Examples\17779\Output\extraction\fs_extract_N.json --layout-json .\Examples\17779\Output\extraction\fs_tick_layout.json --prior-fs-json .\Examples\17779\Output\extraction\fs_extract_N_1.json --output .\Examples\17779\Output\BG_Mapped.xlsx --fs-output .\Examples\17779\Output\FS_Mapped.xlsx --audit-report .\Examples\17779\Output\mapper_audit.md --year 2025

python .\Pipeline\run_document_review.py --current-pdf .\Examples\17779\Input\financial_statements_N.pdf --prior-pdf .\Examples\17779\Input\financial_statements_N_1.pdf --bg .\Examples\17779\Input\bg_standardized.xlsx --output-dir .\Examples\17779\Output

python .\Pipeline\tick_fs_pdf.py --pdf .\Examples\17779\Input\financial_statements_N.pdf --fs-mapped .\Examples\17779\Output\FS_Mapped.xlsx --review-mapped .\Examples\17779\Output\Document_Review.xlsx --output .\Examples\17779\Output\financial_statements_N_ticked_by_pipeline.pdf
```

## Offline Smoke Test With Existing Visual Seeds

This does not call Copilot. It is only for checking that mapping and tickmark
generation still work from existing extraction JSON files.

```powershell
python .\Pipeline\run_pipeline.py `
  --ticket .\Examples\17779 `
  --year 2025 `
  --skip-copilot `
  --seed-fs-json ".\Raw files from service desk\17779\fs_extract_N.json" `
  --seed-prior-json ".\Raw files from service desk\17779\fs_extract_N_1.json" `
  --seed-layout-json ".\Raw files from service desk\17779\fs_tick_layout.json"
```

## Inputs Expected In A Ticket

Default expected files:

```text
Input/
  financial_statements_N.pdf
  bg_standardized.xlsx
  financial_statements_N_1.pdf       optional, used for N-1 tickmarks
```

`fast_reference_do_not_use.xlsm`, if present in an example folder, is only a
reference file. It is not used by the pipeline to generate mappings.
